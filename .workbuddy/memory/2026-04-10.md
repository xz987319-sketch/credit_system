- 1分钟（60秒），在 settings.py 中可配置 SESSION_COOKIE_AGE

## H5申请页验证码校验修复

### 问题
验证码校验逻辑被错误地放在了 `validateFormOnSubmit()` 函数中，但"下一步"按钮是 `type="button"`，走的是独立的 `click` 事件处理器，两套流程互不干涉，导致验证码校验完全不生效。

### 修复
将验证码校验代码（必填校验 + 后端API校验）挪到了 click 事件处理器的正确位置：手机号格式校验（步骤4）之后、金额格式校验（步骤5）之前（约第1407行）。

### 后端校验顺序（verify_sms_code_api）
1. session中是否有验证码记录 → 无 → "请先获取验证码"
2. 手机号一致性 → 否 → "手机号不一致，请重新获取验证码"
3. 验证码正确性 → 否 → "验证码错误"
4. 是否过期（1分钟）→ 否 → "验证码已过期，请重新获取"

## Session残留数据问题（已解决）

### 问题根因
`django_session` 表中有旧的验证码记录（`sms_code_phone='13335678913'`，2026-01-08 创建），导致：
- 用户未获取验证码时，后端 step1 判断失败（session_code 有值）
- 走到 step2 手机号一致性校验，但 session 中的手机号和用户输入不一致
- 结果：即使没获取验证码也报"手机号不一致"

### 解决方案
执行 `python manage.py clearsessions` 清理过期 session，2026-04-10 已执行，清理后 session 表中含 sms_code 的记录数为 0。

### 技术要点
- Django session 存储在 `django_session` 数据库表中，不随 `runserver` 重启清除
- Session 数据字段为 `session_data`（TEXT类型），session_key 是主键
- `clearsessions` 命令只清理已过期的 session（`expire_date < now`），但表中6条记录已全部清理
